## 2. Metrics Inventory

### 2.1 Lambda Runtime Metrics (AWS/Lambda namespace — automatic)

These are emitted by AWS automatically. No code changes needed.

| Metric | Namespace | Dimensions | What it tells you |
|---|---|---|---|
| `Invocations` | AWS/Lambda | FunctionName | Total article processing attempts |
| `Errors` | AWS/Lambda | FunctionName | Failed Lambda runs (uncaught exceptions) |
| `Throttles` | AWS/Lambda | FunctionName | Concurrency limit hits |
| `Duration` | AWS/Lambda | FunctionName | End-to-end processing time per article |
| `ConcurrentExecutions` | AWS/Lambda | FunctionName | Simultaneous Lambda instances |
| `IteratorAge` | AWS/Lambda | FunctionName | S3 event processing lag |

**Grafana Query Example** (CloudWatch data source):
```
Namespace: AWS/Lambda
Metric: Duration
Dimensions: FunctionName = bedrock-development-ai-publisher
Statistic: p50, p90, p99
Period: 300
```

### 2.2 Bedrock Model Invocation Metrics (AWS/Bedrock namespace — automatic)

> [!IMPORTANT]
> These require **Model Invocation Logging** to be enabled in the Bedrock console
> (Settings → Model invocation logging → Enable). Without this, only basic
> `InvocationCount` is available.

| Metric | Namespace | Dimensions | What it tells you |
|---|---|---|---|
| `Invocations` | AWS/Bedrock | ModelId | Total Bedrock API calls |
| `InvocationLatency` | AWS/Bedrock | ModelId | Time Bedrock took to respond |
| `InvocationClientErrors` | AWS/Bedrock | ModelId | 4xx errors (bad requests, throttling) |
| `InvocationServerErrors` | AWS/Bedrock | ModelId | 5xx errors (Bedrock-side failures) |
| `InvocationThrottles` | AWS/Bedrock | ModelId | Rate limit hits |
| `InputTokenCount` | AWS/Bedrock | ModelId | Tokens sent to the model |
| `OutputTokenCount` | AWS/Bedrock | ModelId | Tokens generated by the model |

**Grafana Query Example**:
```
Namespace: AWS/Bedrock
Metric: InputTokenCount
Dimensions: ModelId = anthropic.claude-sonnet-4-6-20260310-v1:0
Statistic: Sum
Period: 86400   # Daily aggregation
```

### 2.3 DynamoDB Metrics (AWS/DynamoDB namespace — automatic)

| Metric | Namespace | Dimensions | What it tells you |
|---|---|---|---|
| `ConsumedReadCapacityUnits` | AWS/DynamoDB | TableName | RCU usage (cost indicator) |
| `ConsumedWriteCapacityUnits` | AWS/DynamoDB | TableName | WCU usage per article write |
| `ThrottledRequests` | AWS/DynamoDB | TableName, Operation | Capacity exceeded |
| `SuccessfulRequestLatency` | AWS/DynamoDB | TableName, Operation | DynamoDB response time |
| `SystemErrors` | AWS/DynamoDB | TableName | DynamoDB internal errors |

### 2.4 S3 Metrics (AWS/S3 namespace — requires request metrics config)

| Metric | Namespace | Dimensions | What it tells you |
|---|---|---|---|
| `NumberOfObjects` | AWS/S3 | BucketName | Total objects (articles + drafts) |
| `BucketSizeBytes` | AWS/S3 | BucketName | Storage cost indicator |
| `AllRequests` | AWS/S3 | BucketName, FilterId | Read/write operations |
| `4xxErrors` | AWS/S3 | BucketName, FilterId | Permission or path issues |

> [!NOTE]
> S3 request-level metrics require enabling "Request metrics" on the bucket
> in the S3 console or via CDK (`bucket.addMetric()`). Storage metrics
> (NumberOfObjects, BucketSizeBytes) are free but reported daily.

## 3. Custom Business Metrics (EMF — Embedded Metric Format)

> [!IMPORTANT]
> **This is a gap.** The Lambda currently logs to `console.log()` but does NOT
> emit structured CloudWatch metrics. Adding EMF logs enables Grafana to query
> business-level KPIs without parsing log text.

### 3.1 What is EMF?

Embedded Metric Format (EMF) lets the Lambda emit custom CloudWatch metrics
by printing structured JSON to stdout. CloudWatch automatically extracts
metrics from the JSON — no SDK calls, no `putMetricData` API, zero latency
overhead.

### 3.2 Proposed Custom Metrics

| Metric Name | Unit | Dimensions | What it tracks |
|---|---|---|---|
| `ArticlesPublished` | Count | Environment, Status | Successful end-to-end publishes |
| `ArticlesFailed` | Count | Environment, FailureStage | Failed at which stage (parse, bedrock, s3, dynamo) |
| `InputTokens` | Count | Environment, Model | Tokens sent to Bedrock per article |
| `OutputTokens` | Count | Environment, Model | Tokens generated per article |
| `ThinkingTokens` | Count | Environment, Model | Thinking budget consumed |
| `EstimatedCostUSD` | None | Environment, Model | Estimated cost per invocation |
| `ProcessingDurationMs` | Milliseconds | Environment, Stage | Time per pipeline stage |
| `DraftSizeBytes` | Bytes | Environment | Input draft size |
| `ContentComplexity` | None | Environment, Level | LOW/MID/HIGH distribution |
| `TechnicalConfidence` | None | Environment | AI self-rated confidence score |
| `RevalidationStatus` | Count | Environment, Status | ISR revalidation success/failure |

### 3.3 EMF Implementation Pattern

Add to the Lambda handler (after Bedrock returns):

```typescript
// bedrock-publisher/src/index.ts — after receiving Bedrock response

// EMF structured log — CloudWatch auto-extracts metrics
console.log(JSON.stringify({
  _aws: {
    Timestamp: Date.now(),
    CloudWatchMetrics: [{
      Namespace: 'BedrockPublisher',
      Dimensions: [['Environment', 'Model']],
      Metrics: [
        { Name: 'ArticlesPublished', Unit: 'Count' },
        { Name: 'InputTokens', Unit: 'Count' },
        { Name: 'OutputTokens', Unit: 'Count' },
        { Name: 'ThinkingTokens', Unit: 'Count' },
        { Name: 'EstimatedCostUSD', Unit: 'None' },
        { Name: 'ProcessingDurationMs', Unit: 'Milliseconds' },
        { Name: 'TechnicalConfidence', Unit: 'None' },
      ],
    }],
  },
  Environment: 'development',
  Model: 'claude-sonnet-4-6',
  ArticlesPublished: 1,
  InputTokens: usage.inputTokens,
  OutputTokens: usage.outputTokens,
  ThinkingTokens: thinkingTokensUsed,
  EstimatedCostUSD: estimatedCost,
  ProcessingDurationMs: endTime - startTime,
  TechnicalConfidence: metadata.technicalConfidence,
  Slug: slug,  // high-cardinality — dimension excluded, searchable in logs
}));
```

**Logs Insights query for recent articles:**
```sql
fields @timestamp, @message
| filter @message like /ArticlesPublished/
| parse @message '"Slug":"*"' as slug
| parse @message '"InputTokens":*,' as inputTokens
| parse @message '"OutputTokens":*,' as outputTokens
| parse @message '"EstimatedCostUSD":*,' as cost
| parse @message '"TechnicalConfidence":*,' as confidence
| sort @timestamp desc
| limit 20
```

## 5. Traces — X-Ray Integration

### 5.1 Current State

| Component | X-Ray Enabled? |
|---|---|
| API Gateway (Bedrock API Stack) | ✅ `tracingEnabled: true` |
| Lambda (ai-publisher) | ❌ **Not enabled** |
| Bedrock Converse API | ❌ No native X-Ray segment |

### 5.2 Required CDK Change — Enable Lambda Active Tracing

```typescript
// infra/lib/stacks/bedrock/ai-content-stack.ts
this.publisherFunction = new lambdaNode.NodejsFunction(this, 'PublisherFunction', {
    // ... existing config ...
    tracing: lambda.Tracing.ACTIVE,  // ← ADD THIS
    // ...
});
```

This gives you:
- End-to-end trace from S3 event → Lambda → Bedrock → DynamoDB → S3
- Subsegments for each AWS SDK call (automatic with X-Ray SDK)
- Latency breakdown per downstream service

### 5.3 Grafana X-Ray Visualisation

The existing Grafana setup does NOT have an X-Ray data source. Two options:

**Option A (Recommended)**: Add an **AWS X-Ray** Grafana data source:
```yaml
# grafana/configmap.yaml — add to datasources
- name: X-Ray
  type: grafana-x-ray-datasource
  access: proxy
  uid: xray
  editable: false
  jsonData:
    defaultRegion: eu-west-1
    authType: default
```

**Option B**: Use the existing CloudWatch data source with the X-Ray
integration (CloudWatch → Traces tab). This works but has fewer features.

### 5.4 Custom Subsegments (Optional)

For more granular tracing, add custom X-Ray subsegments in the Lambda:

```typescript
import AWSXRay from 'aws-xray-sdk-core';

// Wrap each pipeline stage
const bedrockSegment = AWSXRay.getSegment()?.addNewSubsegment('bedrock-converse');
// ... Bedrock call ...
bedrockSegment?.close();

const dynamoSegment = AWSXRay.getSegment()?.addNewSubsegment('dynamodb-write');
// ... DynamoDB write ...
dynamoSegment?.close();
```
